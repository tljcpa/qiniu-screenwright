# Screenwright 开发复盘与路演答辩底稿

> 本文档面向两个用途：一是项目开发的决策日志（为什么这样做、不那样做），二是路演答辩的口播底稿与 Q&A 预案。
> 所有量化数据来自仓库内真实产物 `out/screenplay.yaml` 与端到端冒烟脚本 `scripts/e2e_smoke.py` 的真实运行。
> 论点与 `BRIEF.md`、`docs/schema-design.md`、`backend/app/` 源码严格一致；如有出入以代码为准。

---

## 0. 一句话定位

Screenwright 是一个把小说（3 章以上）转换为结构化剧本（YAML）的智能编剧工作台。
它不是一个"把段落切块再换皮"的转换器，而是一套围绕"剧本即可机读、可手改、可溯源、可重渲染的结构化数据"主张构建的离线编剧工作流。

---

## 1. 选题决策：为什么三题里选题三，为什么不卷算法

### 1.1 为什么是题目三（小说转剧本）

三道赛题里我们明确选了题目三，依据是把"赢面"拆成可控因素后逐项比较的结果：

- **纯文本、零实时脆性，评委现场试用最稳。** 题目三的输入输出都是文本，没有音视频实时流、没有外设依赖、没有现场网络抖动会击穿的环节。评委打开就能跑、跑出来就是确定的产物。相比需要实时语音/视频的题目，这种"演示鲁棒性"在 20% 的演示分上是结构性优势——上一批次我们在语音 demo 录制上踩过坑，知道实时链路在答辩现场有多脆。
- **token 优势只在离线题变现。** 小说转剧本是天然的离线批处理任务：多轮抽取、逐场生成、校验修复，可以从容地多花 token 把质量做厚。实时交互题里 token 预算被延迟卡死，我们手里"可以慢慢算"的优势用不上；离线题才让它变现。
- **Schema 设计文档是白送的架构分。** 赛题硬性要求交付一份 YAML Schema 设计文档（见 `BRIEF.md` 第 0 节必交物料）。题目三的核心产物本身就是结构化数据，写 Schema 文档不是额外负担，而是把我们本就要做的数据契约设计正式化。这份文档（`docs/schema-design.md`）直接落在"过程质量 40%"的架构分上，等于赛题替我们规定了一个能拿满的加分项。

### 1.2 为什么不卷算法

赛题 FAQ 原话写明：评审主依据是**产品价值**，其次才是技术难度。据此我们判定赢的轴是"产品完成度 + 工程规范 + 演示 + 创新决胜"，而不是算法深度，理由：

- **所有原子能力都是 API。** 文本抽取、场景切分、逐场改写，底层全是对 LLM 的调用。没有哪一步是"我们自研了一个比 GPT 更强的模型"——在这种前提下卷算法（自训模型、改 attention）既不现实，也不在评分轴上。
- **评分轴是产品 / 过程 / 演示。** 完整度与创新 40%、过程质量（架构 + 代码 + PR 分布）40%、演示 20%。三条轴没有一条直接奖励"算法难度"。把精力投在算法上是把子弹打在没有靶子的方向。
- **真正的技术含量放在"换皮组会跳过的硬骨头"上。** 我们不卷算法，但绝不等于不做技术。差异化技术含量集中在四件换皮方案做不到的事：跨章一致性、行级溯源、内心戏外化、连贯性检查。这些是工程设计上的硬骨头，不是模型能力的硬骨头——恰好落在过程质量和创新分上。

一句话总结选题策略：**在评委能稳定试用的离线题上，把工程和产品做到换皮组够不到的深度，让 Schema 文档和创新点吃满架构分与创新分。**

## 2. 产品定位：编剧工作台，不是切块转换器

朴素方案是让大模型直接吐一段"看起来像剧本"的纯文本。这在演示层面够用，但丢失了结构，撑不起一个工作台。我们的核心主张是：**剧本应当是一份可机读、可手改、可溯源、可重渲染的结构化数据，纯文本只是它的一种投影。**

围绕这个主张落地四大创新点，每一个都对应一个换皮方案做不到的产品价值：

### 创新点① 可控目标媒介改编

`meta.target_medium` 取 `film | series | short_drama`。同一套 `story_bible` 和场景切分，按不同目标媒介重新渲染出风格不同的剧本——短剧版强调金句、强钩子、情绪密度（中国竖屏短剧打法），电影版更舒缓。
**产品价值**：一次理解原著，多种媒介产出。**差异化**：换皮方案的输出是死的，改媒介得重跑整套理解；我们把"媒介风格"和"故事骨架"解耦，媒介只在生成层作为风格指令注入（见 `generate.py` 的 medium 配置）。

### 创新点② 行级双向溯源

每个内容元素（action / dialogue）带 `source_ref`（章号 + 字符偏移区间）。前端点剧本里某句对白，原文里对应的那几十个字立刻高亮，双向可查。
**产品价值**：改编可信度可验证——作者和评委能精确核对每一行的原文出处。**差异化**：换皮方案给不出"这句台词来自原文哪几十个字"，因为它根本没有保留这个映射。

### 创新点③ 内心戏外化

小说大量依赖内心独白，但剧本拍不了"心里想什么"。我们识别 `interior_monologue`，外化成动作 / 潜台词 / 画外音 / 视觉化，并在该元素上打 `adaptation` 标记（记录 from 什么、用什么 technique）。
**产品价值**：把最容易引起"这是不是篡改原著"疑虑的创造性改写，变成**透明、可审计、可回退**的结构化标注。**差异化**：换皮方案要么直接丢掉内心戏（不忠实），要么硬塞进对白（穿帮）；我们让它显式可控。

### 创新点④ 连贯性检查器

跨场时间线倒错、人物/地点未知引用、重复场景 id、空场景等冲突，由纯规则检查器静态标红（`continuity.py`，确定性、不调 LLM）。
**产品价值**：3 章以上长文本最容易出设定漂移，检查器给作者一张"哪里可能穿帮"的清单。**差异化**：这是结构化数据才做得到的事——纯文本剧本无法做跨场的结构化一致性校验。

## 3. 架构取舍（逐条讲为什么）

### 3.1 离线多轮管线（Pass0-5）而非一次性大 prompt

管线拆成六个 Pass：

```
Pass0 ingest    小说 -> 分章 / 分块（滚动摘要扛长上下文）
Pass1 bible     抽人物 / 地点 / 时间线（跨章一致性单一事实源）
Pass2 segment   场景切分（时空连续）+ 行级 source_ref
Pass3 generate  逐场生成（注入 本场原文 + bible 切片 + 上场尾 + 媒介），内含内心戏外化
Pass4 validate  pydantic 校验每场 + 非法 YAML 自动修复重试 + 连贯性检查
Pass5 export    YAML（主）/ Fountain / PDF
```

**为什么不用一次性大 prompt**：3 章以上的网文很容易超出单次上下文的有效注意力范围，硬塞进去会"中间遗忘"，抽取质量下降；而且让模型同时做抽取 + 切分 + 生成 + 校验是多职责叠加，任一步出错污染全局且不可复现。拆成多轮后，每个 Pass 职责单一、prompt 短而稳，单个 Pass 可独立测试、独立重跑。

### 3.2 elements 用"有序带类型序列"而非按类型分桶（最重要的决策）

直觉但错误的设计是把每场拆成 `actions: []`、`dialogues: []`、`transitions: []` 三个桶。我们明确拒绝，理由：

**剧本的本质是时间序列，顺序本身承载叙事。** "动作—对白—动作—转场"的先后顺序就是导演调度和观众体验的一部分。按类型分桶会抹掉元素间的相对顺序，再也无法表达"先这句台词，再这个动作，然后这句反问"。要在分桶里恢复顺序，就得给每个元素加全局序号，既冗余又脆弱（手改插一行就要重排所有序号）。

因此 `elements` 是单一有序列表，混排三种类型，每个元素自带 `type` 判别字段，pydantic 端用 `Annotated[Union[...], Field(discriminator="type")]` 实现判别联合：按 `type` 直接选子模型校验，错误信息精确、解析更快。这是整套 Schema 里最关键的一处取舍——它让"顺序"成为一等公民，同时服务于"可手改"和"忠实表达叙事节奏"。

### 3.3 story_bible 作单一事实源（分章抽取 + 代码合并，而非一次性丢 LLM）

跨章一致性是刻意啃的硬骨头。如果每个场景各自描述人物、地点，三章下来必然设定漂移。解法是把人物 / 地点 / 时间线抽到顶层 `story_bible` 作为单一事实源，场景内部只用 **id 引用**（`characters: [char_lin]`、`location_id: loc_courtyard`），逐场生成时注入 bible 切片约束模型。一致性问题就从"事后逐场比对"变成"结构上无法分叉"。

**实现上为什么是分章抽取 + 代码合并，而不是把整本书一次性丢 LLM**（见 `bible.py` 的设计注释）：

- **上下文与 token**：分章后每次只看一章，避开长文中间遗忘，抽取更准更全。
- **稳定性与可控**：让 LLM 同时做抽取 + 跨章归一 + 去重 + 分配 id 是多职责叠加，一步抖动污染整张表。把"合并 / 归一 / 分配 id"剥离成纯 Python，确定性算法、同输入同输出、可单测可断点。
- **省 token + 增量友好**：合并不花 token；将来某章被编辑，只需重抽该章再重跑合并。

人物归一用一张 `name/alias -> char_id` 映射表：同一个人在不同章用全名 / 别名 / 简称出现时，命中任意已知键就并入同一 Character，否则新建并分配稳定 id（中文转拼音 slug，冲突追加序号保唯一）。

### 3.4 source_ref 行级下沉 + marker 定位（LLM 只回逐字片段，代码 find 算偏移）

仅有场级溯源只能做到"这一整场来自某章大概哪段"。双向溯源的产品价值在精确到行，所以 `source_ref` 下沉到 element 级，`spans` 用字符偏移（不受原文换行排版影响，前端可字符级精确高亮）。

**关键决策：偏移绝不让 LLM 报，由代码定位**（见 `segment.py` 与 `generate.py` 注释）：

- 大模型按 token 预测，对"第几个字符"没有可靠内部表示。让它直接报 `start=137` 这类偏移几乎一定数错——尤其中文，长文偏差能到上百字符。偏移一错，`chapter.text[start:end]` 取出错位原文，整个行级溯源地基就塌了。
- 正确做法是解耦：**LLM 只做它擅长的语义判断**——切分点在哪、每场开头 / 结尾逐字长什么样（`start_marker` / `end_marker`，10-20 字逐字原文，本质是"抄写"而非"计数"，可靠）；**代码做确定性的定位**——`chapter.text.find(marker)` 内核级精确子串匹配，零误差。
- **兜底契约钉死**：marker 找不到时（模型自作主张改了标点 / 漏字），用"连续接续"兜底——start 找不到从上一场末尾接续，end 找不到延伸到下一场 start 之前。最终保证所有场首尾相接、覆盖整章、每个 `0 <= start < end <= len(text)`，绝不塌成空 span。

### 3.5 schema 即 pydantic 契约 + 校验自动修复环

Schema 不是一份文档，而是 `backend/app/schema/models.py` 里的 pydantic v2 模型——构造即校验。管线各 Pass 围绕同一组类型流转。

`validate_and_repair(raw, llm, max_retries=2)`（`validate.py`）把"LLM 产出不一定合法"这一现实收敛成闭环：统一入口归一成 dict（先试 `json.loads` 报错更精确，失败退 `yaml.safe_load`）→ `Screenplay.model_validate` 触发全量校验 → 失败则把**校验错误文本 + 原始内容**拼成修复对话回喂 LLM，要求只输出修复后的完整 YAML → 重试上限内反复，用尽才抛异常。`llm` 是鸭子类型注入，本模块不绑定具体 client，也不主动发请求。非法不再直接中断管线，而是"非法即回喂修复"。

`Adaptation.from_` 的别名机制（`alias="from"` + `populate_by_name=True`）保证序列化往返一致：`to_yaml()` 用 `by_alias=True, exclude_none=True` 把 `from_` 输出成 `from`、去掉 None 字段，`from_yaml()` 自动回读。`to_yaml -> from_yaml` 构成稳定往返，是"可手改"落地的前提（作者改完导出的文件能原样读回）。

### 3.6 工程方法：架构与契约自定义，AI 辅助编码 + 逐 PR review

开发方式是"我设计、AI 辅助实现、我审校集成"：

- **架构与全局判断由我负责**：数据契约（schema）、各 Pass 的接口与边界、模块依赖关系、测试要求，都是我先定清楚（精确到目标、接口契约、验收标准）。
- **AI 辅助编码、我 review 并集成**：在我定好的契约下用 AI 加速实现各模块；每个改动我审校、对照 schema 校验跑通后才并入。架构取舍与跨模块一致性始终在我手里，AI 不替我做设计决策，最终对结果负责的是我。

**为什么按强依赖链串行推进**：本项目模块间是强依赖链（schema -> llm -> ingest -> bible -> segment -> generate -> validate -> export），多数 Pass 依赖上游产物，只有真正独立、无文件冲突的模块（如 schema 与 llm）才并行实现。每个 diff 经我 review、跑通 pytest 后才算一个 PR，按节奏分布在开发窗口内 push，保证 PR 分布合理（赛题硬约束：禁止最后一天一次性导入）。

| Schema 往返 | 合法（`to_yaml -> from_yaml -> to_yaml` 文本一致） | 工程规范自检 |

- 70.7% 行级溯源覆盖率是**真实的、留白的**——剩下约 30% 是纯外化新增的台词（原文没有对应逐字片段，`source_ref` 诚实地留空 None），不是没做到，而是不该硬凑。

- 15 处内心戏外化全部 from `interior_monologue`，且 action / subtext 两种手法都有，证明外化不是单一套路。
- 0 冲突 + Schema 往返合法，证明产物严格遵守自定义契约，是"工程规范自检"的硬证据。

数据来源说明：覆盖率口径见 `backend/app/pipeline/metrics.py`（溯源分母只算 action + dialogue，排除转场；原文覆盖用区间合并去重防虚高）。

## 5. 踩坑与解决

### 5.1 marker 定位 vs 让 LLM 报偏移

- **坑**：最初想让 LLM 直接输出每段的字符偏移 start/end。模型对"第几个字符"无可靠内部表示，中文长文偏差到上百字符，取回的原文严重错位，行级溯源直接失效。
- **解**：把"定位"还给代码。LLM 只复制 10-20 字逐字 marker（抄写它擅长），代码 `text.find(marker)` 精确算偏移；marker 找不到时用连续接续兜底，保证 spans 覆盖整章、不越界、`start < end`。这是整个溯源机制能成立的根。

### 5.2 PDF 的 CJK 字体子集化

- **坑**：fpdf2 内置 Helvetica 是 Latin-1 编码，遇到中文直接抛 `UnicodeEncodeError`，PDF 导出会崩。
- **解**（`export.py` 的 `to_pdf`）：探测系统 CJK 字体候选路径（文泉驿 / Noto / 思源），命中则 `add_font` 注册为 `cjk` 族全文用它（fpdf2 默认 uni=True，会按用到的字形做 Unicode 子集嵌入，PDF 不会塞进整套字体）；找不到字体或字体解析失败则降级内置 Helvetica，并对非 latin-1 字符做安全处理，保证导出在任何环境都不崩——宁可降级也不让 demo 现场炸。

### 5.3 章内偏移自洽设计

- **坑**：如果 `source_ref` 用全文偏移，分章后一旦某章增删，下游所有偏移全部漂移，溯源失效。
- **解**（`types.py` 的设计点）：所有溯源偏移一律相对**该章 `chapter.text`**（章内偏移），不相对全文。理由有二：章内偏移只依赖该章自身，章节增删不影响其他章；前端本就按章展示原文，章内偏移可直接 `text[start:end]` 定位。E2E 脚本把"`chapter.text[start:end]` 非空且自洽"作为创新点②的根基不变式，逐场断言校验。

### 5.4 LLM 产出不一定合法

- **坑**：LLM 偶尔吐出非法 YAML / 缺字段 / 字段越界，直接 `model_validate` 会中断整条管线。
- **解**：`validate_and_repair` 把校验错误文本回喂 LLM 自修复，重试上限内闭环；统一入口先 JSON 后 YAML 容错解析。非法不再是终止态，而是一次修复机会。

## 6. 路演 Q&A 预案

**Q1：三道题你们为什么选小说转剧本这道？**
A：选离线纯文本题，评委现场试用最稳——没有实时音视频会击穿的脆性环节，打开就能跑、产物确定。而且离线题让我们"可以慢慢多花 token 把质量做厚"的优势变现，赛题又硬性要交 Schema 文档，而我们的核心产物本就是结构化数据，写 Schema 文档等于赛题替我们规定了一个能拿满的架构加分项。

**Q2：底层都是调 API，技术含量在哪？**
A：原子能力确实是 API，但难度不在模型能力，在工程设计。我们做了四件换皮方案做不到的事：跨章一致性单一事实源、行级双向溯源、内心戏外化的透明标注、跨场连贯性检查。比如行级溯源——让 LLM 报字符偏移它一定数错，我们让 LLM 只复制逐字 marker、代码 find 算偏移，把不确定的语义判断和确定的数值计算解耦。这类取舍才是技术含量所在。

**Q3：elements 为什么是一个混合列表而不是按类型分桶？**
A：剧本本质是时间序列，元素的先后顺序本身就是叙事——"先台词、再动作、再反问"。分桶会抹掉相对顺序，要恢复就得加全局序号，冗余又脆弱（手改插一行就要重排）。所以我们用判别联合（discriminated union），单一有序列表混排，每个元素带 type 字段，pydantic 按 type 直接选子模型校验，错误精确、扩展也只需加子模型不动旧数据。这是整套 Schema 最关键的取舍。

**Q4：行级溯源覆盖率只有 70.7%，是不是没做完？**
A：恰恰相反，这 70.7% 是诚实的。剩下约 30% 是纯外化新增的台词——原文根本没有对应的逐字片段，我们让 source_ref 诚实留空 None，而不是硬塞一个错误的出处。如果硬凑到 100%，反而是在制造假溯源。同时原文覆盖率 86.3%，证明剧本忠实用到了原著绝大部分内容。

**Q5：跨章一致性怎么保证的？为什么不把整本书一次喂给模型？**
A：我们把人物 / 地点 / 时间线抽到顶层 story_bible 作单一事实源，场景内部只用 id 引用，生成时注入 bible 切片——一致性从"事后比对"变成"结构上无法分叉"。不一次性喂全书有三个原因：长文中间遗忘导致抽取质量下降；让模型同时抽取 + 归一 + 去重 + 分 id 是多职责叠加、一步抖动污染整表；而我们把合并 / 归一 / 分 id 剥成纯 Python，确定性、可单测、还省 token、增量友好。

**Q6：内心戏外化会不会算篡改原著？**
A：这正是我们做 adaptation 标记的原因。剧本拍不了"心里想什么"，必须外化成动作 / 潜台词 / 画外音，这是创造性改写。我们把每一处外化显式标注——from 什么、用什么 technique——所以它透明（评审一眼看出哪些是忠实转写、哪些是主动改编）、可审计、可一键筛出全部外化元素做对照甚至回退。我们把最容易被质疑的环节变成了最可控的环节。

**Q7：开发过程踩的最大的坑是什么？**
A：让 LLM 报字符偏移。模型按 token 预测，对"第几个字符"没有可靠表示，中文长文偏差上百字符，取回的原文全错位，行级溯源地基就塌了。解决办法是责任倒置：LLM 只复制逐字 marker（它擅长抄写），代码用 find 精确定位，找不到时用连续接续兜底保证覆盖整章不越界。次大的坑是 PDF 中文字体——fpdf2 内置字体是 Latin-1，遇中文直接崩，我们探测系统 CJK 字体注册并做子集嵌入，找不到就安全降级，保证 demo 现场不炸。

**Q8：你们的 PR 是怎么组织的，怎么保证过程质量分？**
A：我先定数据契约和各模块接口，再在契约下用 AI 辅助实现、自己 review 每个 diff、跑通 pytest 才并入。模块间是强依赖链（schema -> llm -> 管线各 Pass），所以串行为主，只有真正独立的模块才并行。每个模块配 pytest，关键断言走 schema 校验，每个 diff 我 review 通过才算一个 PR，按节奏分布在开发窗口内 push，不在最后一天一次性导入。
